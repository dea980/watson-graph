# watson-graph 로드맵 — 금융 도메인 v2 + Next UI

두 트랙이 있다. **엔진**(타입드 관계 그래프)과 **프론트**(Next SPA). 두 트랙은 API로 만나고, **eval**이 엔진 효과를 숫자로 증명한다. 순서는 병행이 아니라 **기능+데이터(백엔드) 먼저, UI 나중**으로 직렬화한다(아래 순서 원칙). 설계 근거: `docs/GRAPHRAG-DOMAIN-DESIGN.md`.

## 트랙

**T1 — 그래프 엔진 (co-occurrence → 타입드 관계)**
- A. `EdgeType` 도입, `adjacency` 타입드 확장, 시드 관계표, BFS 경로 계산. 하위호환 테스트 그린 유지.
- B. 관계 추출기 — 패턴 규칙 확장(`~에 따라`=GOVERNED_BY 등), 시드표 적용. 단위 테스트로 타입 정확도.
- C. DART 실데이터 적재 — 공개 API 커넥터, 구조화 메타에서 ISSUES/DISCLOSES 직접. 기업(ISSUER) 노드.
- D. eval 변별 픽스처 — 코퍼스 30~50개 + 하드네거티브 + vector-hard golden. 타입드 on/off A/B로 효과 증명.

**T2 — 프론트엔드 (Next.js App Router, TS)**
- U1. 스캐폴딩 + 일반 RAG 화면(금융 페르소나). **기존 API만 소비 — 엔진 의존 없음.**
- U2. 그래프 경로 시각화 화면(타입드 멀티홉을 노드/엣지로). **T1-A 경로 반환 API에 의존.**
- U3. 배포 — jar 통합(static export) 또는 분리(Vercel+백). 빌드 파이프라인 확정.

**완료된 선행 작업** (2026-06-17)
- 엔진 패치 2건: HybridRetriever `graph.weight`, 문서 단위 `linkDocument`(청크 경계 멀티홉 복구). 단위테스트 8/8 그린.

## 의존성

```
T1-A 엔진 ──▶ T1-B 추출기 ──▶ T1-D eval(증명)
   │                              ▲
   └──▶ U2 경로 시각화            │
                                  │
T1-C DART 적재 ───────────────────┘   (보류 트랙)

U1 일반 화면 ── 독립(지금 바로) ──▶ U3 배포
```

- U1은 의존 없음 → 지금 시작.
- U2는 T1-A(경로 반환)와 설계 결정 3b(경로 노출) 필요.
- T1-D는 T1-A/B가 있어야 lift를 보여줌. 현재 eval은 코퍼스 8개 천장이라 그 전엔 무의미.
- T1-C(DART)는 스키마가 시드/픽스처로 검증된 뒤. 인증·네트워크 변수가 검증을 흐리지 않게 분리.

## 순서 원칙

**기능과 데이터를 먼저 확정하고, UI는 그다음.** UI는 API 소비자라 백엔드가 런타임에 살아있고 데이터가 그래프까지 제대로 적재돼야 의미가 있다. 그래서 단계를 트랙 병행이 아니라 백엔드 우선으로 직렬화한다.

## 진행 현황 (2026-06-17)

- 백엔드 E2E 확인됨 — `/api/rag/ask`가 멀티홉(사업보고서→자본시장법→파생결합증권) 정답을 출처 근거와 함께 반환. Ollama 임베딩 + ingest + RAG 살아있음. kr-securities count=8.
- T1-A 타입드 엔진 — 코드 완료(`EdgeType`+시드표, 타입드 adjacency, `searchWithPaths` 경로 반환). 로컬 빌드/테스트 그린 대기.
- graph-stats 관측 — `GET /api/data/graph/stats?namespace=` 추가(노드/엣지/타입별 분포). 그래프·타입드 적재 검증용.
- U1(FE) — Next 스캐폴딩 완료, **파킹**. P2에서 연결.
