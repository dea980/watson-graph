# 멀티모달 지식화 & OCR Grounding

> 이미지를 검색 가능한 거버넌스 지식으로 바꾸는 방법과, 그 한계까지 기록한다.
> **실제 테스트에서 관찰한 사실**만 적었다. 실패 사례가 핵심이다.

---

## 1. 무엇을 하는가

기존 data·ai·governance 루프 위에 두 가지 기능을 얹었다.

| 엔드포인트 | 목적 |
|---|---|
| `POST /api/multimodal/ask` | 이미지 + 질문 → grounded 답변 (단발 시각 Q&A) |
| `POST /api/multimodal/ingest` | 이미지를 `Article`로 저장 → 이후 **텍스트** RAG 질문으로 검색 가능 |

둘 다 기존 부품을 그대로 재사용한다 — `EmbeddingService`, `VectorIndex`,
`ArticleParquetStore`, governance 감사 로그. **새 layer는 추가하지 않았다.**
멀티모달은 기존 3개 layer를 *가로지른다*:

```
이미지 업로드
  ├─ Vision 모델(llava)이 레이아웃/맥락 설명   → AI Layer  (watsonx.ai)
  ├─ OCR(Tesseract)이 정확한 텍스트/숫자 추출   → AI Layer  (전처리)
  ├─ 캡션+OCR을 임베딩해 Parquet에 저장          → Data Layer (watsonx.data)
  └─ 모델·latency·프롬프트 로깅                  → Governance (watsonx.governance)
```

---

## 2. 왜 추출기가 둘인가 (Vision **그리고** OCR)

처음엔 vision 모델 **하나로만** 이미지를 읽었다. 숫자에서 처참하게 실패했다.

- 송장 총액을 물으니 `llava`가 **`$650`** 이라고 답함 — 송장에 없는 숫자. 순수 환각.
- 막대차트에서 최고 매출 분기를 물으니 완전히 지어낸 표를 만듦
  (`$2M Q1'20, $1M Q2'20, $5M Q3'21, $3M Q4'21`) — 연도도 값도 다 틀림.

**결론:** 비전 언어 모델은 이미지를 *설명*("이건 라인아이템 표가 있는 송장")하는
건 잘하지만, *정확한 값을 읽는* 건 못 믿는다. 자신만만하게 숫자를 지어낸다.

그래서 역할을 분리했다:

| 추출기 | 담당 | 이유 |
|---|---|---|
| **OCR** (Tesseract) | 정확한 텍스트·숫자 | 인쇄 문자에 대한 결정론적 인식 |
| **Vision** (llava) | 레이아웃·문서종류·맥락 | 의미 이해 (정확한 읽기는 아님) |

두 결과를 합쳐서 저장한다:

```
[OCR]
<Tesseract가 읽은 원문>

[Vision]
<이미지가 무엇인지 한두 문장 설명>
```

Vision 프롬프트는 **숫자 읽기를 명시적으로 금지**하도록 바꿨다:

```
"이 이미지의 레이아웃과 종류(차트/표/문서/사진)와 무엇에 관한 것인지
 한두 문장으로 설명하라. 구체적인 숫자는 읽거나 추측하지 마라 —
 숫자 값은 OCR이 따로 추출한다."
```

그리고 RAG 증강 프롬프트가 어느 소스를 믿을지 명시한다:

```
"아래 context를 사용하라. 정확한 숫자는 [Vision] 설명보다 [OCR] 섹션을 신뢰하라."
```

### Before/After (같은 송장, 같은 질문 "송장 총액은?")

| 구성 | 답 |
|---|---|
| Vision만 | `$650` (환각) |
| Vision+OCR, vision이 여전히 숫자 읽음 | `$5M`/오답 (자신만만한 vision 텍스트를 LLM이 믿음) |
| Vision(레이아웃만)+OCR+"OCR 신뢰" 프롬프트 | **`$852.50`** |

이 진화 과정이 핵심 교훈이다: **소스를 합치는 것만으론 부족하다 — 어느 소스가
권위인지 모델에게 알려줘야 하고, 못 믿는 소스가 경쟁하지 못하게 막아야 한다.**

## 3. 한계 (이론이 아니라 실제 관찰)

OCR이 환각 문제는 해결했지만 자기만의 실패 모드를 가져왔다. 모두 테스트에서 재현됨.

### 3.1 저대비 / 반전 텍스트

첫 테스트 송장은 TOTAL을 **검은 박스 + 흰 글씨**로 렌더링했다. Tesseract가 깨뜨림:

```
TOTAI $85          ← 실제로는 "TOTAL $852.50"
```

같은 송장의 나머지(Subtotal `$775`, Tax `$77.50`, 라인아이템 전부)는 완벽하게
읽었고 — **반전된 그 칸만** 실패했다. 그래서 최종 답은 자신만만한 **`$85`**:
파이프라인은 *OCR을 올바르게 신뢰했지만*, OCR이 틀렸다.

- **원인:** Tesseract는 밝은 배경의 어두운 글씨에 맞춰져 있다. 흰 글씨/검은 배경,
  저대비, 꾸민 셀에서 급격히 나빠진다.
- **먹힌 해결책:** TOTAL을 밝은 배경의 어두운 글씨로 다시 렌더링 → OCR이
  `TOTAL $852.50`을 정확히 읽었고 답도 정답이 됨.
- **실전 입력용 완화책:** OCR 전 이미지 전처리(grayscale → 대비 스트레칭 →
  적응형 이진화, 그리고 반전 감지). 실제 문서 AI 시스템이 인식 전에 돌리는 게 이거다.

### 3.2 OCR은 2D 구조를 잃는다 (차트)

막대차트에서 Tesseract는 값과 축 라벨은 읽었지만, **막대와 분기의 공간적 매핑이
사라진 평면 텍스트 스트림**으로 읽었다:

```
$30M $20M $10M $0       (y축 눈금)
$12M $19M $28M          (막대 값 — Q3의 $15M은 아예 누락)
Q1   Q2   Q3   Q4       (x 라벨, 일부 깨짐: "Ql ... a4")
```

`$28M`(진짜 최댓값)이 OCR 텍스트에 있긴 했지만, **그게 `Q4`에 속한다는 정보는
선형 텍스트 어디에도 없다.** 평면 OCR 덤프로는 "어느 분기가 최고?"를 신뢰성 있게
답할 수 없다 — 막대↔라벨 연결이 사라졌기 때문.

- **원인:** OCR은 1D 문자 인식이다. 차트는 2D 위치·색·막대 높이로 의미를
  인코딩하는데, 이 중 무엇도 평면 OCR엔 남지 않는다.
- **함의:** 표(행 구조)는 OCR이 잘 되고, 차트/플롯은 안 된다. 차트엔 차트 전용
  파서, 레이아웃 인지 문서 모델, 또는 차트 이해에 특화된 강한 비전 모델이 필요하다
  (예: IBM Granite Vision은 차트/표/KVP 추출을 노리지만, 여기선 2B Ollama 빌드가
  너무 작았고 테스트 중 런너 크래시까지 났다).

### 3.3 소형 비전 모델은 사실 추출에 못 쓴다

`granite3.3-vision:2b`는 송장 질문에 대시 한 줄(`-----`)만 냈고, 별도로 Ollama
런너를 크래시(`exit status 2`)시켰다 — Ollama 업데이트 전까지. `llava:7b`는
돌았지만 숫자를 환각함(2절). 실전 교훈: **모델 크기와 런타임 성숙도가 지배적이다** —
어떤 모델도 쓸만한 답을 내기 한참 전에 데이터 배관은 이미 맞았다.

## 4. 여기서 도출된 설계 원칙

1. **"읽기"와 "이해"를 분리하라.** 사실은 결정론적 추출기(OCR), 맥락은 생성 모델.
   생성 모델이 못 읽는 사실을 지어내게 두지 마라.
2. **권위 순서를 선언하라.** 여러 소스를 합칠 때, 충돌 시 어느 소스가 이기는지를
   프롬프트가 명시해야 한다 (`[OCR]을 [Vision]보다 신뢰`).
3. **정직하게 실패하라.** context에 정말 답이 없으면, 지어내지 말고 거절하도록
   유도한다 — 매출 데이터 없는 지식베이스에 매출 질문이 왔을 때 실제로 거절함을 확인.
4. **어려운 건 모델이 아니라 파이프라인이다.** 프로젝트의 핵심 명제와 같다 —
   모델 선택은 쉬운 부분이고, 전처리·소스 융합·grounding에서 정확도가 갈린다.

## 5. 향후 과제

- **OCR 전처리**: Tesseract 전에 grayscale, 대비/이진화, 반전 감지 (3.1절을 원본
  이미지 수정 없이 해결했을 것).
- **차트 이해**: 레이아웃 인지/차트 특화 모델 통합, 런타임 안정화되면 더 큰
  Granite Vision 빌드 평가.
- **구조화 추출**: vision/OCR 단계가 자유 텍스트 대신 JSON
  (`{"total": 852.50, "lineItems": [...]}`)을 내게 해서, 숫자를 타입 있는 쿼리
  대상으로.
- **신뢰도 신호**: 필드별 OCR confidence를 저장하고, 낮은 값은 governance 로그에
  표시해 검토 유도.

### 5.1 진짜 멀티모달 임베딩 (CLIP) — 추후 계획

현재는 "이미지 -> OCR+Vision -> 텍스트 -> 단일 텍스트 임베딩 공간(nomic)"이다. 즉
**가난한 자의 멀티모달**: 이미지 내용을 텍스트로 환원해 기존 공간에 넣는다. 차트·표·스캔
문서 같은 엔터프라이즈 이미지는 대부분 이걸로 충분하다.

진짜 멀티모달(CLIP류)은 이미지와 텍스트를 **같은 벡터 공간에 직접 임베딩**해 "이 사진과
시각적으로 비슷한 것"까지 검색한다. 도입을 미루는 이유와 경로:

- **왜 무겁나(난이도가 아니라 인프라)**: Ollama는 CLIP식 공동 임베딩을 제공하지 않는다
  (llava는 캡션은 내지만 임베딩 벡터는 아님). 별도 임베더 런타임이 필요하다 —
  Python 사이드카(open_clip/sentence-transformers), DJL(arm64 네이티브 확인 필요),
  또는 클라우드 API. Ollama-only 단순함이 깨진다.
- **최소 변경 경로**: `nomic-embed-vision-v1.5`는 `nomic-embed-text-v1.5`와 **같은 공간**으로
  정렬돼 있어, 텍스트를 그 버전으로 바꾸면 이미지·텍스트가 **한 인덱스**에 공존한다
  (VectorStore 추상화 덕에 인덱스는 그대로, 임베더만 추가).
- **가치 판단**: 순수 시각 유사도가 필요할 때만 ROI가 있다. 엔터프라이즈 RAG는
  OCR+Vision->텍스트로 80%가 잡히므로, 시각 검색 요구가 구체화되기 전엔 보류한다.
- **연관**: PDF 내부 이미지도 현재는 임베딩되지 않는다(텍스트 레이어만). CLIP을 도입하면
  PDF 이미지 추출 -> 공동 임베딩으로 함께 다룰 수 있다.

## 6. API 레퍼런스

### 시각 Q&A
```bash
curl -X POST http://localhost:8080/api/multimodal/ask \
  -F "image=@invoice.png" \
  -F "question=What is the invoice total?" \
  -F "model=llava:latest"          # 선택; 없으면 ollama.vision-model 기본값
```
응답:
```json
{ "answer": "$852.50", "model": "llava:latest" }
```

### 이미지를 지식베이스에 ingest
```bash
curl -X POST http://localhost:8080/api/multimodal/ingest \
  -F "image=@invoice.png" \
  -F "namespace=demo" \
  -F "model=llava:latest"          # 선택
```
응답: 저장된 `Article` (`summary`에 `[OCR]` + `[Vision]` 텍스트 포함). 이후
이미지 내용은 일반 텍스트 RAG로 검색된다:
```bash
curl -X POST http://localhost:8080/api/rag/ask \
  -H "Content-Type: application/json" \
  -d '{"question":"What was the invoice total?","namespace":"demo"}'
```

> **참고:** chat 모델은 요청별로 바꿀 수 있지만, **임베딩 모델은 지식베이스당
> 고정**이다 (임베딩 모델을 섞으면 유사도가 깨짐). OCR은 best-effort —
> Tesseract가 없거나 실패하면 vision 캡션만으로 ingest가 계속된다.
