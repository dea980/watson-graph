# watson-graph — 용량 · 확장 설계 (Capacity & Scaling)

> "몇 명이 쓸 수 있나? 200~500, 나아가 공공·금융 기관 ~1,000명을 보고 설계됐나?"에 대한 정직한 분석과 확장 경로.
> 결론 먼저: **현재는 단일 노드 연구/데모 아키텍처로, 특정 사용자 수를 목표로 용량 설계되지 않았다.**
> 병목은 검색이 아니라 **로컬 LLM 생성**이며, 핵심 변수는 "등록 인원"이 아니라 "동시 요청 수(concurrency)"다.
>
> **공공·금융 ~1,000명 목표라면:** 등록 1,000은 소수 노드로 가능하나, ① 피크 동시 요청 50~100을 받아줄 **LLM 워커 풀**과 ② 엔터프라이즈 요건(**폐쇄망·보안·감사·HA·컴플라이언스**)을 별도로 충족해야 한다. 다행히 **로컬 LLM이라 폐쇄망(망분리)에 적합**하고, 테넌트 격리·PII 거버넌스·API키 인증이 이미 있다(§6.5, §7, §8).

---

## 1. 시스템 개요와 이번에 추가된 것

watson-graph = **MiniWatson 전체 + GraphRAG(증권·공시) 레이어**.

추가된 것
- `KnowledgeGraph` — 도메인 엔티티 공출현 그래프 + 멀티홉(BFS 깊이 2) 탐색
- `DomainGlossary / DomainEntityExtractor / QueryRewriter` — 증권·자본시장(공시·IR) 사전, 엔티티 추출, 약어↔정식명 질의 재작성
- `HybridRetriever` — 벡터(의미) + BM25(어휘) + **그래프(관계)** 를 RRF로 융합
- 토글: `retrieval.graph.enabled`, `persona.domain.enabled`, 요청 `graph`/`hybrid`/`rerank`(EVAL)

### 페르소나
RagService의 `DOMAIN_PERSONA`:
> 자본시장·기업공시(DART) 전문 어시스턴트. 공시·재무제표·자본시장법에 근거해 답하고, 수치·일정·법적 효력은 보수적으로 다루며 추정으로 단정하지 않음. 근거 없으면 "모른다"고 답하고 출처(문서 제목)를 제시. 투자 판단은 본인 책임이며 투자자문이 아님.

성격: **증권/공시 도메인 + 규정·출처 근거 + 보수적·면책** 톤.

## 2. 요청 1건의 처리 경로와 비용

`POST /api/rag/ask` 한 건의 내부 비용:

| 단계 | 작업 | 대략 비용 | 확장성 |
|---|---|---|---|
| 질의 재작성 | 문자열 처리(LLM 0회) | < 1 ms | 무관 |
| 임베딩 | Ollama `/api/embed` 1회 | 수십 ms | 중간(임베딩 서버) |
| 검색 | 벡터+BM25+그래프(전부 인메모리) + RRF | 수~수십 ms | CPU/메모리 |
| 리랭크 | MMR(인메모리 코사인) | 수 ms | CPU |
| **생성** | **Ollama `/api/generate` 1회** | **수 초** | **★ 핵심 병목** |

→ **검색부는 매우 빠르다(그래프 추가해도 ms 단위).** 한 요청 시간의 90%+는 **LLM 생성**이 차지한다. 따라서 용량은 사실상 **LLM 처리량**으로 결정된다.

---

## 3. "등록 인원"과 "동시 요청"은 다르다

- **등록(총) 사용자 N** — 큰 의미 없음. 대부분 idle.
- **동시 요청(concurrency) C** — 실제로 같은 순간 LLM 생성을 기다리는 요청 수. **이게 용량을 결정한다.**

경험적으로 사내 도구는 **동시 요청 ≈ 활성 사용자의 1~5%** 수준이다. 즉 등록 500명이라도 피크 동시 요청은 보통 **5~25건** 정도다(사용 패턴에 따라 변동).

---

## 4. 처리량 산정 (가정 명시)

가정 (하드웨어·모델에 따라 크게 달라짐 — 반드시 실측 필요):
- 생성 응답 길이 `num-predict = 256` 토큰
- 단일 GPU에서 모델 처리 속도 ≈ 40~80 tok/s → **생성 지연 L ≈ 3~6초/요청**
- 단일 Ollama 인스턴스 동시 생성 슬롯 `G ≈ 1~3` (모델·VRAM 의존)

**노드 처리량 ≈ G / L**
- 예: G=2, L=4s → **0.5 req/s ≈ 30 req/분 ≈ 1,800 req/시간**

체감 지연은 동시 요청 C가 슬롯 G를 넘으면 큐 대기로 급격히 악화된다(C ≤ G일 때 쾌적).

## 5. 시나리오별 수용 인원 (단일 노드 1대 기준, 위 가정)

| 상황 | 판단 |
|---|---|
| 등록 200~500명, **피크 동시 요청 ≤ 2~3** | ✅ 단일 GPU 박스 1대로 충분 |
| 등록 200~500명, 피크 동시 요청 10~30 | ⚠️ 큐 지연 발생 → LLM 슬롯/노드 증설 필요 |
| **등록 ~1,000명(공공·금융), 피크 동시 50~100** | ⚠️ LLM 워커 풀 2~6노드 + 무상태화 + HA 필요 (§7) |
| **동시 요청(접속이 아닌 실제 호출) 200~500+** | ❌ 단일 Ollama 불가 → 수평 확장 필수 |

요지: **"등록 인원"이 1,000이어도 한~소수 노드로 가능**하지만, **"동시 요청"이 수십~수백이면 LLM을 그만큼 늘려야 함.** 기관 사용은 업무시간 집중·버스트가 있어 피크 동시성을 보수적으로 잡아야 한다.

## 6. 현재 아키텍처의 확장 한계

- 그래프·벡터·키워드 인덱스가 **단일 JVM 인메모리** → 수직 확장만, 노드 간 공유 불가(지금 상태로 수평 확장 안 됨)
- 저장: H2 + Parquet 단일 노드 (벡터만 pgvector 옵션 존재)
- LLM/임베딩이 **단일 Ollama**에 직접 동기 호출 — 큐·백프레셔 없음
- 레이트리밋·오토스케일·분산 캐시 없음
- 재시작 시 인메모리 인덱스는 Parquet/DB에서 재적재(hydrate)

→ 이 모든 게 "단일 노드 데모"라는 원래 전제와 일치한다. **200~500 동시는 비목표였다.**

## 6.5 공공 · 금융 엔터프라이즈 고려사항 (~1,000명)

기관 사용은 "동시성"만의 문제가 아니라 **운영·보안·컴플라이언스** 요건이 함께 온다. 현재 가진 자산과 보강점:

| 요건 | 현재 자산 | 보강 필요 |
|---|---|---|
| **폐쇄망/망분리(온프레미스)** | ✅ 로컬 Ollama — 외부 API 호출 없음(폐쇄망 적합) | 컨테이너 이미지·모델 가중치 내부 레지스트리 배포 |
| **접근통제·인증** | ✅ API키 필터 + 테넌트 격리 | SSO/LDAP·SAML 연동, 역할기반 권한(RBAC) |
| **감사·추적성** | ✅ QueryLog(질문·답변·지연·출처) + PII 마스킹 | 변경불가 감사로그 보존, SIEM 연계 |
| **데이터 격리** | ✅ namespace 멀티테넌시 | 기관/부서 단위 분리, 보존·파기 정책 |
| **가용성(HA)** | ❌ 단일 노드 | 무상태화 + 다중 인스턴스 + DB 이중화 |
| **암호화** | 부분 | 전송(TLS)·저장(at-rest) 암호화, 키 관리 |
| **컴플라이언스** | 일부(거버넌스 토대) | 전자금융감독규정·개인정보보호·공공 망분리지침 매핑 |

핵심: **로컬 LLM + 기존 거버넌스/테넌트 격리는 공공·금융과 본질적으로 잘 맞는다.** 빈 칸은 주로 **HA·인증연동·암호화·컴플라이언스 매핑**이다.

## 7. 동시 50~500 요청을 목표로 한 목표 아키텍처

```
            ┌── 무상태 App 노드 (N개, 오토스케일) ──┐
 LB / API ─►│  질의재작성·검색·RRF·리랭크·프롬프트   │─► 큐 ─► LLM 워커 풀
 Gateway    └──────────────────────────────────────┘        (vLLM/TGI ×M)
   │             │ 공유 스토어                                  │
   │             ├─ 벡터: pgvector / Qdrant                     │
   │  레이트리밋  ├─ 그래프: Neo4j                              └─ 연속 배칭으로 동시성↑
   └─ 캐시(Redis) └─ 임베딩 서버(별도 풀)
```

핵심 변경 4가지
1. **LLM을 풀로** — Ollama 다중 인스턴스 또는 **vLLM/TGI(연속 배칭)** + 로드밸런서. *가장 큰 비용·병목.* 연속 배칭은 노드당 동시성을 수십까지 끌어올린다.
2. **앱 무상태화** — 벡터(pgvector/Qdrant)·그래프(Neo4j)를 외부 공유 스토어로 분리 → 앱 노드를 N대로 수평 확장.
3. **요청 큐 + 백프레셔 + 레이트리밋** — 피크 흡수, 폭주 시 우아한 저하(429/대기).
4. **캐시 + 스트리밍** — 동일/유사 질의 캐시, 토큰 스트리밍으로 체감 지연 완화.

### LLM 노드 수 산정(러프)
- vLLM 1노드가 연속 배칭으로 동시 요청 `Cn ≈ 16~32` 처리 가정 시
- 동시 요청 목표 C → **LLM 노드 수 ≈ ⌈C / Cn⌉**
  - C=50 → 2~3노드, **C=100(공공·금융 ~1,000명 피크) → 4~7노드**, C=200 → 7~13노드, C=500 → 16~32노드 (모델·길이·GPU 의존, 실측 필수)

## 8. 단계별 로드맵

| 단계 | 목표 | 핵심 작업 |
|---|---|---|
| **Phase 0 (현재)** | 등록 ~수백, 저동시(≤3) | 단일 노드. 그대로 사용 가능 |
| **Phase 1 (앱 내, 인프라 무증설)** | 피크 동시 10~30 안정화 | **LLM 동시성 게이트+큐+429**, 응답 캐시, 테넌트 레이트리밋, actuator 메트릭 |
| **Phase 2 (수평 확장)** | 공공·금융 ~1,000명 / 동시 50~100 | LLM을 vLLM 워커 풀로 + 앱 무상태화(pgvector+Neo4j) + LB + 캐시 + 오토스케일 |
| **Phase 3 (엔터프라이즈 하드닝)** | 공공·금융 운영 | HA 이중화, SSO/LDAP·RBAC, TLS/at-rest 암호화, 감사로그 보존·SIEM, 컴플라이언스 매핑 |

## 9. 실측 방법 (추정 대신 숫자 확보)

```bash
# 단일 노드 한계 측정: 동시 c를 올려가며 p95 지연/처리량 관찰
#   k6 / wrk / hey 등으로 /api/rag/ask 부하
hey -n 200 -c 4  -m POST -T application/json \
  -d '{"question":"공매도가 이뤄지는 시장의 대표 주가지수는?","namespace":"kr-securities"}' \
  http://localhost:8080/api/rag/ask
```
- `c`(동시)를 1→2→4→8…로 올리며 **p95 지연이 무너지는 지점 = 단일 노드 동시 슬롯 G**.
- 그 G와 평균 지연 L로 노드 처리량(G/L)을 확정한 뒤, 목표 동시 C로 노드 수를 역산한다.

## 10. 가정 · 면책

- 위 수치는 **모델·GPU·응답 길이에 따라 수 배 차이**가 난다. 설계 전 반드시 자체 환경에서 §9로 실측할 것.
- 검색부(그래프 포함)는 병목이 아니며, 데이터가 커지면 인메모리 → pgvector/Neo4j 외부화가 먼저다.
- 비용의 대부분은 **LLM/GPU**다. 동시성 목표가 곧 GPU 예산이다.
- 본 문서는 아키텍처 용량 가이드이며, 특정 SLA·비용을 보증하지 않는다.
